# SALJIP DB Search Coverage Full Audit (2026-03-31)

## 목적
- "`DB에 없어서 검색이 안 되는 케이스` + `DB에는 있는데 검색식 때문에 안 잡히는 케이스`"를 전체 DB 기준으로 점검.
- 대상: 현재 허브 검색의 핵심 테이블 `complex` 전체.

## 실행 방법
- 스크립트: `notes/tmp_db_audit.cjs`
- 결과 파일: `notes/hub-search-db-audit-2026-03-31.json`
- 추가 검증(키워드 커버리지): ad-hoc SQL 조회

## 핵심 수치 (전체 DB)
- 총 단지 수: `8,258`
- 정규화 쿼리 공백(빈 문자열) 케이스: `5,382`
- 정규화 기반 self-match 실패: `609`
- 공백 제거 기반 self-match 실패: `92`

해석:
- 현재 검색식에서 "문자 정규화 방식"이 약하거나 과도한 부분이 있고, 특히 특수문자/공백/브랜드 표기 변형에서 누락이 발생할 수 있음.
- `fail_no_space`가 `92`인 점은, "공백/기호 무시 매칭"만 보강해도 즉시 줄일 수 있는 실패군이 있다는 의미.

## 사용자 이슈 직접 검증: `고덕아이파크`
- 정규화 exact 매칭 건수: `0`
- 이름 포함 매칭(`%고덕%` + `%아이파크|ipark%`) 건수: `0`
- `legal_dong`에 `고덕`/`상일` 포함 건수: `0`
- `apt_name`에 `고덕` 포함 건수: `0`

결론:
- 현재 DB 기준으로는 `고덕아이파크` 계열 데이터가 **미적재 상태**라 검색 실패가 발생하는 것이 맞음.
- 즉, 이 케이스는 검색 UI/알고리즘 문제가 아니라 데이터 커버리지 문제.

## 발견된 구조적 원인
- 원인 A (커버리지): 특정 생활권/단지군 미적재.
- 원인 B (검색식): 브랜드/공백/특수문자 변형에 대한 허용 범위가 제한적.
- 원인 C (UX): 결과 없음일 때 "DB 미커버리지" vs "검색식 미일치"가 사용자에게 구분되지 않음.

## 우선순위 액션
1. 데이터 우선 보강
- `고덕/상일` 포함 누락 지역부터 ingest backfill.
- "검색량 높은데 결과 0" 키워드 로그 기반으로 주간 보강.

2. 검색식 보강
- `no-space` 정규화 컬럼 기반 매칭 추가.
- 브랜드 alias 사전 적용: `아이파크/IPARK`, `e편한세상/e편한`, `더샵/THE SHARP` 등.
- 질의어 토큰 AND 매칭(예: `고덕` + `아이파크`) 추가.

3. UX 보강
- 결과 0일 때 안내 분기:
  - `DB 미수록 가능성` 문구
  - 유사 키워드 자동 제안
  - "요청 단지 등록" CTA (백로그 연결)

## 상태
- 전수조사(전체 DB 수치 + 사용자 이슈 키워드 직접 검증) 완료.
- 다음 단계는 `검색식 보강` + `누락 커버리지 backfill` 병행이 정답.
